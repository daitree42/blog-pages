用 20 行 Node.js 为 DeepSeek 修复 Claude Code 的 system 消息 400 错误

2026年5月29日 3 分钟
Claude Code DeepSeek Node.js 代理 教程

把 Claude Code 接到 DeepSeek 的 V4 Flash 模型上，价格便宜量又足，这是很多开发者的首选方案。但跑起来之后，时不时就会迎面砸来一个 400 错误，说消息格式不对。

仔细一看，Claude Code 在发起请求时，会在 messages 数组里塞多条 role: "system" 的消息——CLAUDE.md 一条、记忆文件一条、当前日期一条……而这些在 DeepSeek 的 Anthropic 兼容接口上不被接受，直接报格式错误。

解决思路很直接：写一个本地代理，在请求到达 DeepSeek 之前，把多余的系统消息"翻译"一下。

思路

Anthropic 的对话协议允许 messages 数组中出现多条 role: "system" 的消息，但 DeepSeek 的兼容实现只认一条。最简单的修复是：

  • 保留第一条 system 消息不动
  • 将其余的 system 消息改为 role: "user"，并在内容前加上 [System Note]: 前缀

这样 DeepSeek 能正常解析，而信息本身没有丢失——只是从"系统指令"变成了"用户消息中的系统注记"，对模型的理解影响微乎其微。

实现

一个不到 60 行的 Node.js 脚本，监听本地 4000 端口，做三件事：

  1. 接收 Anthropic 格式的请求
  2. 清洗消息数组中的冗余 system 消息
  3. 转发到 DeepSeek 官方接口

代码很直白：

const http = require('http');
const https = require('https');

const PORT = 4000;
const DEEPSEEK_HOST = 'api.deepseek.com';

const server = http.createServer((req, res) => {
  let body = '';
  req.on('data', chunk => body += chunk);
  req.on('end', () => {
    let payload = body;

    if (body && req.url.includes('/v1/messages')) {
      try {
        const parsed = JSON.parse(body);
        if (Array.isArray(parsed.messages)) {
          let sysCount = 0;
          parsed.messages = parsed.messages.map(msg => {
            if (msg.role === 'system') {
              sysCount++;
              if (sysCount > 1) {
                return { ...msg, role: 'user',
                  content: `[System Note]: ${msg.content}` };
              }
            }
            return msg;
          });
          payload = JSON.stringify(parsed);
        }
      } catch (_) {}
    }

    const proxy = https.request({
      hostname: DEEPSEEK_HOST,
      path: req.url,
      method: req.method,
      headers: { ...req.headers, host: DEEPSEEK_HOST,
                 'content-length': Buffer.byteLength(payload) },
    }, proxyRes => {
      res.writeHead(proxyRes.statusCode, proxyRes.headers);
      proxyRes.pipe(res);
    });

    proxy.on('error', e => {
      res.writeHead(502);
      res.end('Bad Gateway: ' + e.message);
    });

    proxy.write(payload);
    proxy.end();
  });
});

server.listen(PORT);

核心逻辑其实就一个 map 加一个计数器——第一条 system 跳过，后续的全部改 role 加前缀。请求和响应都是流式管道直通，没有额外延迟。

使用方式

保存为 proxy.js，然后：

# 启动代理（后台挂起）
node /root/proxy.js &

# 设置环境变量，让 Claude Code 走代理
ANTHROPIC_BASE_URL="http://127.0.0.1:4000" claude

为了让日常使用更方便，我在 ~/.bashrc 里加了一个函数，让默认的 claude 命令自动拉起代理：

claude() {
  (node /root/proxy.js &>/dev/null &) 2>/dev/null
  ANTHROPIC_BASE_URL="http://127.0.0.1:4000" command claude "$@"
}

# 同时也留一个直连通道备用
alias claude-direct='ANTHROPIC_BASE_URL="https://api.deepseek.com/anthropic" claude'

效果

加上这层代理之后，400 错误彻底消失，所有 Claude Code 的功能——CLAUDE.md 指令注入、记忆文件读取、工具调用——全部正常工作。

代理的开销几乎为零：只是改了两个 JSON 字段名就原样转发，不缓存、不记录、不修改响应体，耗时在微秒级别。

小结

这是一个很典型的"兼容层补丁"场景——两个 API 协议在核心功能上一致，但在边缘细节上存在差异。与其等上游修复，不如在中间加一层轻量的胶水代码。20 行逻辑，一个 map，解决问题。

如果你想在自己的环境里用，把上面的代码保存为 proxy.js，修改 DEEPSEEK_HOST 为你自己的 API 地址即可。整件事的核心不是什么高深技术，而是知道问题出在哪。
