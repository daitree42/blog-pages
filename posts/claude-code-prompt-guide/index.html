Claude Code Prompt 编写指南

2026年5月25日
Claude Code Prompt

一、核心认知：Claude Code 不是聊天机器人

Claude Code 是一个 agentic loop（代理循环）系统。这意味着：

聊天式 Claude Claude Code
你问一句它答一句 你给一个任务，它自主执行多步
只能输出文字 可以读文件、写代码、执行命令
被动回应用户 主动探索代码库、制定计划、执行操作
单轮对话 多步循环（思考→行动→观察→再思考）

所以 prompt 的核心变化是： 你不是在"提问"，而是在"分配任务"。不需要一步步指导它怎么做，只需要说清楚目标，它会自己规划路径。

二、三种交互模式

1. Approval Mode（审批模式，默认）

Claude 每执行一个操作都先征求你的同意。提示词不需要特殊处理，Claude 会自动停下来等你确认。

2. Auto-Accept（自动接受模式）

Claude 可以直接执行命令和写文件，不需要逐个确认。适合你信任的、低风险的任务。

3. Plan Mode（计划模式）

Claude 只做规划不写代码。先用 prompt 让它制定方案，你确认后再执行。

如果你不确定怎么做，先进入 Plan Mode 让我看方案。

或者在启动时用 /plan 命令。

三、Prompt 的结构化方法

一个好的 Claude Code prompt 通常包含以下要素：

1. 角色 / 上下文（Context）

告诉 Claude 它在什么项目里、面对什么场景。

我在一个 Next.js 博客项目中，使用了 Tailwind CSS。

2. 目标（Goal）

清晰描述你要达成的结果，而不是具体步骤。

❌ "帮我打开 src/pages/index.tsx，找到第 42 行，把那个 button 的颜色从 blue 改成 red"
✅ "把首页的按钮颜色从蓝色改成红色"

后者更好，因为 Claude 会自己去找到文件、分析代码、做正确的修改。你告诉它"做什么"，它自己知道"怎么做"。

3. 约束条件（Constraints）

如果有特殊的限制，说清楚。

不要修改 API 层的代码。
使用现有的 UI 组件库，不要引入新的依赖。

4. 输出期望（Expectations）

如果有特定的输出要求，提出来。

先给出方案再执行。
执行完后总结改了哪些文件。

四、好 prompt 与差 prompt 的对比

差 prompt

帮我改一下这个页面

— 太模糊，Claude 不知道该做什么。

好 prompt

在用户个人资料页面，把"保存"按钮移到页面右上角，点击后显示一个 toast 提示"保存成功"。

约束：使用现有的 Button 和 Toast 组件，不要新增文件。

— 明确目标 + 约束，Claude 可以直接执行。

差 prompt

优化这个 API

— 目标不清晰，"优化"可以是速度、可读性、安全性……

好 prompt

分析 app/api/products.ts 中的查询性能，找出响应慢的原因并修复。优先检查数据库查询是否有 N+1 问题。

— 具体问题 + 排查方向。

差 prompt

搜索代码里所有的 TODO 并处理

— 范围太大，可能改到不该改的地方。

好 prompt

搜索 src/utils/ 目录下的 TODO 注释，逐个列出给我看，我决定哪些要处理。

— 先探索再决策。

五、结合工作流的 Prompt 策略

对应 Explore → Plan → Code → Commit，每个阶段的 prompt 写法不同：

Explore 阶段：聚焦信息采集

帮我了解这个项目的路由结构，列出所有页面文件。
重点是 pages/ 目录下的文件，不要看 components/。

Plan 阶段：聚焦方案

我想加一个搜索功能，支持按标题和标签过滤文章。
先在 Plan Mode 下出方案，列出涉及的文件和改动点。

Code 阶段：聚焦执行

按刚才的方案，实现搜索功能。先加后端查询接口，再改前端页面。

注意：前面已经讨论过方案了，所以这里不需要再重复所有细节。

Commit 阶段：聚焦总结

提交当前改动，commit message 要说明改了什么、为什么改。

六、复杂任务的 Prompt 技巧

1. 分步推进，不要一口吃成胖子

第一步：帮我分析 src/auth/ 下面的代码结构，画一个简单的说明。
# 等他回复后，再给第二步
第二步：在 auth service 里加一个登出接口，清除 token。

2. 使用「if 思维」写 prompt

如果你不确定某些文件的作用，先列出来问我，不要猜。

如果某个操作会删除文件，先停下来问我确认。

3. 明确"不要做什么"

不要改动测试文件。
不要格式化我没有指定的文件。
不要升级依赖版本。

4. 给出例子（Few-shot）

我需要给所有的 API 路由加错误处理。格式参考已有的 getUser 接口：
try {
  // ...
} catch (error) {
  console.error('操作失败:', error);
  throw new Error('操作失败，请稍后重试');
}

七、常见错误

错误 说明 正确做法
太模糊 "帮我优化一下" 说清楚优化什么、标准是什么
步骤太多 一次给 10 个子任务 拆成 2-3 个批次
不说约束 Claude 改了不该改的 提前说好不要动什么
假设 Claude 记得 以为它还记得之前的上下文 关键信息反复说，或用 CLAUDE.md
把 Claude 当搜索引擎 "React 18 有什么新特性" Claude Code 不适合搜文档，用 WebSearch

八、实战模板

功能开发类

在 [项目] 中，实现 [功能描述]。

要求：
- 使用现有的 [组件/工具]
- 不要修改 [特定文件/目录]
- [其他特殊约束]

先出方案，我确认后再写代码。

Bug 修复类

[描述 bug 现象]

复现步骤：
1. [步骤 1]
2. [步骤 2]

预期行为：[应该怎么样]
实际行为：[实际怎么样]

帮我定位原因并修复。

代码审查类

review 当前的改动，重点关注：
1. 有没有类型错误
2. 有没有遗漏的边缘情况
3. 有没有安全风险

只在有问题的行上提 comment。

九、总结：写好 prompt 的三条原则

  1. 说"什么"不说"怎么" —— 告诉 Claude 目标，让它自己决定路径
  2. 先窄后宽 —— 从小范围开始，Claude 证明能力后再扩大任务
  3. 关键信息前置 —— 项目背景、约束条件、不要做什么，放 prompt 开头
