搭了一个零维护的静态博客

2026年5月25日
博客 静态站点 Python

一、为什么

一直在本地写文章，但写完的东西都躺在文件夹里，不方便分享。想搭一个博客，但有几条硬约束：

  • 长期不维护也不会坏。不用框架，不用数据库，不用任何需要定期升级的东西
  • 不改变现有的写作流程。文章还是用 Markdown 写，存在同一个目录结构里
  • 零成本。不买服务器，不交月费
  • 中文排版要好。读起来舒服，不花哨

翻了一圈市面上的方案，静态站点生成器很多（Hugo、Jekyll、Hexo），但都依赖特定的运行时生态。五年后这些工具的版本可能已经面目全非，而我不想去折腾升级。

于是决定自己写一个。

二、方案

核心思路就一条：用纯 Python 把 Markdown 转成 HTML，推到 GitHub Pages 上。

整个博客是一个 Python 脚本 + 三个 Jinja2 模板 + 一个 CSS 文件，放在 /blog/ 目录下。运行时只需要系统自带的 Python 3 和两个 apt 包（python3-markdown、python3-jinja2），没有任何需要 node_modules 或 gemfile 的东西。

博客目录/
├── publish.py         # 核心生成器
├── deploy.sh          # 构建 + 部署
├── templates/         # Jinja2 模板
│   ├── base.html
│   ├── index.html
│   └── post.html
├── static/
│   └── style.css      # 中文排版样式
└── dist/              # 构建输出，推送到 Pages

文章解析

文章的目录结构保持了写作时的原样：

articles/<slug>/
├── draft.md           # 正文（Markdown）
├── outline.md         # 大纲
├── notes/             # 采访笔记
└── sources/           # 参考资料

publish.py 遍历 articles/ 目录，从每个 draft.md 中解析元数据和正文。元数据写在标题后的 blockquote 里：

# 文章标题

> 栏目：Ai技术
> 日期：2026-05-25
> 标签：标签1，标签2
> 摘要：一段简短的描述

正文从这里开始……

用 blockquote 做元数据的好处是：即使解析器出了问题，它在 Markdown 渲染后看起来仍然是一段正常的引用文字，不会破坏页面。

构建流程

publish.py 做的事情很简单：

  1. 扫描所有文章目录，解析 draft.md
  2. 用 python3-markdown 把正文转成 HTML
  3. 填入 Jinja2 模板，生成首页和文章页面
  4. 把模板、样式、静态资源复制到 dist/

整个脚本不到 200 行。

部署

构建完成后，dist/ 目录就是一个完整的静态站点。deploy.sh 把这个目录推送到 GitHub Pages 仓库，Pages 自动托管——不需要自己配 Nginx，不需要管 HTTPS 证书，不需要担心服务器宕机。

还加了一个监控脚本，用 inotifywait 监听 draft.md 的修改，检测到变动后自动构建部署。这样写完文章保存文件，博客就自动更新了。

三、设计上的几个选择

不用前端框架

一个静态博客不需要 React 或 Vue。纯 HTML + CSS 的页面加载最快，且不会因为依赖升级而突然不能用了。CSS 里用了衬线字体栈、合适的行距（1.9）和字号（18px），针对中文阅读做了优化。

路径用子目录

GitHub Pages 的项目站点托管在 username.github.io/repo-name/ 下，路径里多一级子目录。模板中所有链接都用 site_prefix 变量控制，切换根路径或自定义域名只需改一个配置项。

保留 .git

每次构建会清空 dist/ 目录，但保留了 .git，这样每次部署都是增量推送，不用重新初始化仓库。

封面图可选

如果有 cover.jpg 放在文章目录下，构建时会自动复制到页面目录。没有也不影响，属于渐进增强。

四、感受

写这个生成器的过程比预期的愉快。一方面是因为需求很明确，不需要纠结"万一以后要加评论怎么办"（答案是真的需要再说）；另一方面是因为纯 Python 加模板的方案几乎没有心智负担，改起来很快。

相比于用现成的博客框架，自己写生成器的好处是完全可控。想加一个栏目分类页面，改几行代码就行。想调整 RSS 格式，直接改 XML 模板。没有插件系统的学习成本，也没有框架版本的迁移焦虑。

当然，它缺的东西也很明显：没有评论区（目前用不着），没有搜索功能（文章多了再加），没有统计分析（我也不想看）。一个博客最核心的东西还是内容本身，工具只要能稳定地把文字发布出去就够了。

这种极简的体感很好。工具应该隐身，而不是刷存在感。

五、结果

现在这个博客就运行在这个生成器上。写完这篇文章，保存文件，监控脚本检测到变动，十秒后站点自动更新。

整个系统依赖列表：

  • Python 3（系统自带）
  • python3-markdown（apt 安装）
  • python3-jinja2（apt 安装）
  • GitHub Pages（免费托管）

没有 Docker，没有 node_modules，没有数据库，没有 Nginx 配置。理论上十年后回来，装两个 apt 包就能重新跑起来。

这大概就是"零维护"的最终含义——不是不用管，而是想管的时候，也不必操什么心。
